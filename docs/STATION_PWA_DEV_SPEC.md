## 2. Jobs to Be Done (JTBD) Analysis

### 2.1 Primary Job

> **"When** I'm assigned to run a distribution station in a disaster zone,
> **I want to** track incoming supplies and outgoing distributions,
> **So that** I can serve evacuees effectively and report back to HQ."

### 2.2 Job Breakdown

| # | Job | Trigger | Outcome |
|---|-----|---------|---------|
| J1 | **Setup Station** | First-time use or new assignment | Station is paired with Hub, ready to operate |
| J2 | **Receive Supplies** | Runner arrives with manifest QR | Inventory updated, receipt confirmed |
| J3 | **Distribute Supplies** | Evacuee needs supplies | Supplies given, transaction logged |
| J4 | **Register Person** | New evacuee arrives | Person recorded in local system |
| J5 | **Report to Hub** | Shift end or Runner arrives | QR report generated for transport |
| J6 | **Check Status** | Anytime | View current inventory & pending sync |

### 2.3 Scenario Matrix

| Scenario | Actor | Frequency | Connectivity |
|----------|-------|-----------|--------------|
| Morning setup | Station Lead | 1x/day | May have WiFi briefly |
| Receive supplies | Station Lead + Runner | 2-3x/day | Offline |
| Distribute | Station Lead + Evacuee | 10-50x/day | Offline |
| Register person | Station Lead + Evacuee | 5-20x/day | Offline |
| End-of-day report | Station Lead + Runner | 1x/day | Offline |

## 4. UI/UX Specifications

### 4.1 Design Principles

| Principle | Implementation |
|-----------|----------------|
| **Clarity First** | Large buttons with clear labels + descriptions |
| **Offline Visible** | Always show sync status, offline banner |
| **Error Prevention** | Confirm before destructive actions |
| **Flat Design** | Teal + Deep Blue palette, no gradients |
| **Heroicons Only** | No emoji in UI, use heroicons outline |

### 5.2 Hub Pairing Protocol

**QR Format for Pairing**:
```json

  "version": "2.0",

  "hub_encryption_key": "base64..."   // X25519 encrypt

```

**Pairing Flow**:
1. Hub Admin creates Station in dashboard
2. Hub generates pairing QR
3. Station scans QR â†’ stores config
4. Station ready to operate

### 5.3 Data Sync Protocol

**Outbound (Station â†’ Hub)**:
1. Collect pending actions from IndexedDB
2. Build `REPORT_PACKET` with HMAC
3. Encrypt with `hub_encryption_key`
4. Chunk into QR codes (800 bytes max)
5. Display for Runner to scan
6. Mark as "in transit" (not yet synced)

**Inbound (Hub â†’ Station)**:
1. Scan manifest QR
2. Verify Ed25519 signature
3. Check if manifest already processed
4. Apply inventory changes
5. Record manifest as processed

## 6. API Dependencies

### 6.1 Hub API (Backend)

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `POST /api/logistics/stations` | Create | Register new station |
| `GET /api/logistics/stations/{id}/pairing-qr` | Read | Get pairing QR |
| `POST /api/logistics/manifests` | Create | Create manifest |
| `POST /api/logistics/reports` | Create | Receive station report |
| `GET /api/logistics/stations/{id}/status` | Read | Check sync status |

### 6.2 Local Storage (IndexedDB)

- reports         // Generated reports (history)
